# 三人行吊顶管理系统 - 系统架构文档 V4.0

## 1. 系统概述

三人行吊顶管理系统是一个用于管理吊顶工程、工资结算、预支管理、AI智能辅助的综合性系统。

### 1.1 技术栈

| 层级 | 当前 | 规划 |
|------|------|------|
| 前端 | Vue 3 + TypeScript + Vant UI | **Android原生APK** |
| 后端 | Node.js + Koa + PostgreSQL | 同左 |
| AI | - | 通义千问/文心一言/ChatGLM/DeepSeek/豆包 |
| 部署 | Linux服务器 + PM2 | 同左 |

## 2. 权限体系 V2.0

### 2.1 角色定义

| 角色 | 代码 | 定位 |

| 管理员 | admin | 只读监管 + 系统配置，**不干预业务操作** |

| 资料员 | documenter | 查看全部工程(只读) + 新建工程 |

## 3. 核心模块

### 3.1 用户管理模块
- 用户注册、登录、权限管理
- 角色：admin（系统配置+只读监管）、documenter（查看+新建）、constructor（业务操作）

### 3.2 工程管理模块
- 工程创建、编辑、状态管理
- 子项目管理
- Service层: `services/projectService.js`（规划）
- Repository层: `repositories/projectRepo.js`（规划）

### 3.3 结算管理模块
- 工资结算（施工员自确认，无需管理员审批）
- 预支管理
- 结算历史

### 3.4 统计模块
- 工程统计、金额统计、月度报表
- Redis缓存层（规划）

### 3.5 AI模块（新增）
- **排料引擎**：本地计算面材/龙骨/配件需求
- **大模型调用**：支持通义千问/文心一言/ChatGLM/DeepSeek/豆包
- **知识库RAG**：施工规范、材料参数、安全标准检索
- **Function Calling**：AI按用户权限查询工程/统计/结算数据

## 4. 设计原则

### 4.1 数据一致性原则
- 工程状态与子项目状态同步
- 结算状态通过视图动态计算，不存储冗余数据
- 金额计算在后端完成，前端只负责展示

### 4.2 权限隔离原则 (V2.0)
- 管理员（admin）：系统配置 + 全局查看，不干预业务操作
- 资料员（documenter）：查看所有工程详情 + 新建工程
- 施工员（constructor）：对自己参与的工程有完全操作权

### 4.3 状态机原则
- 工程状态：preparing → constructing → completed → canceled
- 结算状态：unsettled → settling → settled

### 4.4 独立结算原则

系统采用**用户独立结算**设计，每个用户的结算是完全独立的，互不影响。

- 只获取当前结算用户的预支记录
- 只为当前结算用户创建结算快照
- 只更新当前结算用户的结算状态
- 结算流程：施工员自确认，无需管理员审批

### 4.5 排料引擎原则 (AI新增)
- 材料参数只存在知识库，引擎不硬编码
- 排料计算纯本地(<50ms)，不依赖网络
- 大模型只做分析和建议，不参与核心计算
- SVG数据在后端生成，前端接收即渲染

## 5. 关键约束

1. **创建工程时**：必须计算金额，必须更新工程总额
2. **工程完工时**：必须同步更新子项目状态为completed
3. **结算时**：施工员只能结算自己参与且已完工的工程
4. **AI查询时**：自动注入用户权限过滤
5. **排料计算时**：从知识库动态加载材料参数
