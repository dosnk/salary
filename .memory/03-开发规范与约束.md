## 💰 2. 金额计算——硬约束

⚠️ **绝对禁止前端参与金额计算。**

| 规则 | 实现方式 | 位置 |
|-----|---------|-----|
| 所有金额由后端计算 | `POST /v1/settlements/calculate` 是唯一接口 | [controllers/settlements.js](file:///f:/AIPoject/salary/backend/controllers/settlements.js) |
| 金额字段类型 | `NUMERIC(14,4)`（PostgreSQL）→ 前端映射为 Double | 数据库 Schema + [ProjectDtos.kt](file:///f:/AIPoject/salary/salary-android/core/network/src/main/java/com/salary/core/network/dto/ProjectDtos.kt) |
| 前端显示 | 使用 `AmountFormatter.format(Double?)`，保留两位小数 | Android `core/design` 模块 |
| 金额计算必须在事务内 | 子项目 amount 先计算再 INSERT，禁止事后 UPDATE | [services/projectService.js](file:///f:/AIPoject/salary/backend/services/projectService.js) + [repositories/projectRepo.js](file:///f:/AIPoject/salary/backend/repositories/projectRepo.js) |
| 工程总额重算 | 子项目增删改后必须调用 `recalculateProjectTotal(projectId)` | [services/projectService.js](file:///f:/AIPoject/salary/backend/services/projectService.js) |

**金额计算引擎唯一入口**：
```javascript
services/calculation.js
  ├── calculateSubprojectAmount(unit, length, width, price)   // 子项目金额
  └── calculateProjectTotal(projectId)                        // 工程总额
```

**违反后果**：前后端金额不一致 → 用户投诉 → 需要回滚数据库。

## 🗓️ 3. 时区与日期处理——硬约束

⚠️ **所有按月份筛选的查询必须处理时区。**

| 场景 | 规则 | 正确写法 | 错误写法（禁止） |
|-----|-----|---------|---------------|
| 年月份筛选（WHERE） | 必须用 `AT TIME ZONE 'Asia/Shanghai'` | `TO_CHAR(p.created_at AT TIME ZONE 'Asia/Shanghai', 'YYYY-MM')` | `TO_CHAR(p.created_at, 'YYYY-MM')` |
| 显示（仅显示用） | 可以按 UTC，但不推荐 | `TO_CHAR(p.created_at, 'YYYY-MM-DD HH24:MI')` | — |
| day 级筛选 | 通常无需时区处理（偏差较小） | `TO_CHAR(p.created_at, 'YYYY-MM-DD')` | — |

**必须应用的位置**：
- [projectRepo.js](file:///f:/AIPoject/salary/backend/repositories/projectRepo.js) 的 `listQuery` 和 `countQuery`
- 任何使用 `WHERE TO_CHAR(created_at, 'YYYY-MM') = '2026-06'` 的查询

**违反后果**：当天（00:00 ~ 08:00 CST = 前一天 UTC）创建的工程会被计入"上月"。

## 🔒 4. 安全规则

| 规则 | 详情 |
|-----|-----|
| SQL 注入防护 | **所有 SQL 必须参数化**，使用 `$1, $2, $3...` 占位符，禁止字符串拼接。搜索关键词：`` ` `` `+` `template literals` 拼接用户输入 |
| JWT Token | access_token 过期 2h，refresh_token 30d。401 时 Android 端自动刷新。 |
| 密码强度 | 最小 6 位 + 字母数字组合。 |
| 登录失败锁定 | 连续失败 5 次锁定 30 分钟。 |
| API 限流 | 按用户/IP 频率限制。 |
| 敏感数据脱敏 | 密码、Token 等敏感数据输出日志前需脱敏。dataMasking.js 已实现。 |

## 👤 5. 权限体系（RBAC v2）

### 角色定义
| 角色 | 权限范围 | 定位 |
|-----|---------|-----|
| **admin** | 用户管理 + 全部工程查看 + 字典管理 + AI 配置 + 数据迁移 + 统计 | **只读监管 + 系统配置**，不修改工程内容 |
| **constructor** | 自己参与的工程 CRUD + 结算 + 预支 + 自己的统计 | **业务操作用户** |
| **documenter** | 全部工程只读 | **文档员**，不做任何修改 |

### 权限检查位置
| 层级 | 实现 | 说明 |
|-----|-----|-----|
| 路由层 | [middleware/rbac.js](file:///f:/AIPoject/salary/backend/middleware/rbac.js) | 粗粒度角色检查 |
| 业务层 | service 层 + `permissionFilter.js` | 细粒度资源检查（如"只能修改自己参与的工程"） |
| 前端按钮 | `v-permission` 指令（Vue）+ role 判断（Compose） | 不显示无权限按钮（仅是体验优化，后端仍需校验） |

**关键约束**：
- `admin` **不能创建工程**、**不能修改工程**、**不能操作结算**。
- 登录后的页面默认是 `constructor` 视角。
- AI 对话数据自动过滤：admin 看全部，constructor 看自己的工程数据。

## ❌ 10. 已废弃 / 禁止使用的模式

| 旧模式 | 新模式 | 原因 |
|-------|-------|-----|
| `projects.settlement_status` 字段 | `project_user_status` 表（按用户维度） | 一个工程多人可分别结算 |
| 前端本地计算金额 | `POST /v1/settlements/calculate` | 避免前后端不一致 |
| 前端硬编码材料单价 | 从数据库 `construction_plans` 表读取 | 动态配置 |
| `AuthInterceptor` 401 直接清除 token | 401 先尝试 refresh，失败再清 token | 减少不必要的重新登录 |
| 纯 UTC 时间筛选 | `AT TIME ZONE 'Asia/Shanghai'` | 适配中国用户时区 |
| `v-permission='admin'` 放行所有操作 | 仍需后端 rbac.js 校验 | 前端仅作按钮隐藏 |

## ✅ 11. API 响应规范

后端统一使用 ApiResponse Wrapper：
```json
{
  "code": 200,           // 200 = 成功，其他 = 错误码
  "message": "success",  // 中文提示
  "data": { ... }        // 响应体（可为 null / 数组 / 对象）
}
```

Android 端对应：
```kotlin
@Serializable
data class ApiResponse<T>(
    val code: Int,
    val message: String,
    val data: T? = null
)
```

前端错误消息由 `NetworkErrorHandler.translateServerError()` 翻译为中文：
- `"数据库异常"` → `"服务器数据异常，请联系管理员"`
- `token / 认证 / 授权` → `"登录已过期，请重新登录"`
- `权限 / forbidden` → `"您没有权限执行此操作"`
- `参数错误` → `"请求参数有误，请检查后重试"`
- JSON 反序列化异常 → `"数据格式异常，请更新应用版本"`
