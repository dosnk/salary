## 👥 角色与权限体系（V2.0）

三种角色，权限严格分离，不可越级操作：

### 1. admin（管理员）
- ✅ 用户管理：全部功能
- ✅ 工程查看：全部工程
- ✅ 字典管理：空间类型、施工方案
- ✅ 数据迁移：数据库初始化与迁移
- ✅ AI 助手配置：大模型提供商选择、API Key 配置
- ✅ 统计查看：全部数据
- ❌ 工程创建/修改/删除、结算操作、预支管理（仅可查看）
- **定位**：只读监管 + 系统配置，不参与业务操作

### 2. constructor（施工人员）
- ✅ 工程管理：创建、查看、修改自己参与的工程
- ✅ 子项目管理：添加、编辑、转交手项目
- ✅ 结算操作：查看结算单，提交结算
- ✅ 预支管理：创建、查看自己的预支记录
- ✅ 统计查看：自己参与工程的统计数据
- ✅ 站内消息：接收工程状态变更、结算完成等通知

### 3. documenter（文档员）
- ✅ 工程查看：全部工程（只读）
- ❌ 工程创建/修改/删除、结算、预支、用户管理

## 🔑 核心业务概念

| 概念 | 说明 |
|-----|-----|
| **工程 (Project)** | 一个客户地址对应一个工程，包含多个子项目 |
| **子项目 (Subproject)** | 工程下的具体施工任务：空间类型 × 施工方案，有独立的长宽、数量、金额 |
| **空间类型 (SpaceType)** | 如：客厅、卧室、厨房、卫生间、阳台（字典表，可配置） |
| **施工方案 (ConstructionPlan)** | 如：封梁、装饰条、扣板、铺砖等（含单价和计量单位，字典表） |
| **施工人员 (Worker/User)** | 工程的参与人员，工资按参与度分配 |
| **工资分配 (Salary Distribution)** | 平均分配 (average) 或按工日分配 (work_days) |
| **预支 (Advance)** | 施工人员提前支取的金额，结算时扣除 |
| **结算 (Settlement)** | 工程完成后计算应付金额 = 工程总金额 − 已预支金额 |
| **工程状态** | preparing(备料中) / constructing(施工中) / completed(已完工) / canceled(已取消) |
| **结算状态** | unsettled(未结算) / settling(统计中) / settled(已结算) |

> **重要设计决策**：工程状态（全局统一，客观事实）与结算状态（用户独立，个人行为）分离。
> - `projects.status` 表示工程的全局施工状态
> - `project_user_status.settlement_status` 记录每个用户在每个工程中的独立结算状态

## 🔄 核心业务流程

### 工程创建流程
```
输入客户地址
  → 选择空间类型（客厅/卧室/厨房...）
  → 选择施工方案（封梁/装饰条/扣板...）
  → 输入长度、宽度（厘米）
  → 选择参与施工人员（1 人或多人）
  → 选择工资分配方式（平均/按工日）
  → 系统自动调用 [calculation.js](file:///f:/AIPoject/salary/backend/services/calculation.js) 计算金额
  → 创建工程 + 子项目 + 更新工程总额（事务内完成）
  → 记录工程历史
```

### 结算流程
```
选择工程 → 查看结算预览（后端计算）
  → 提交结算请求
  → 后端生成结算单记录（扣减预支）
  → 更新 `project_user_status` 表中的结算状态为 settled
  → 发送站内消息通知
  → 工程列表中显示"已结算"
```

### 工程状态流转
```
备料中 (preparing)
     ↓
施工中 (constructing)   ←→ 正常施工
     ↓
已完工 (completed)     → 所有参与人员结算状态自动置为 settling
     ↓
已取消 (canceled)      → 保留历史，不参与结算
```

## 📂 主要数据实体关系

```
users (用户表)
  └── id, username, password(加密), nickname, role, phone, created_at

projects (工程表)  核心
  ├── id, name(客户地址), description, status, total_amount, created_by, created_at
  │
  ├── project_workers (施工人员关联表)
  │     ├── project_id, user_id, work_days
  │
  ├── subprojects (子项目表)
  │     ├── id, project_id, space_type_id, construction_plan_id, length, width, quantity, amount, created_by
  │
  ├── files (附件表)
  │     ├── id, project_id, filename, path, size, uploaded_by
  │
  ├── project_history (修改历史)
  │     ├── id, project_id, action, description, performed_by, created_at
  │
  └── project_user_status (用户结算状态表) 【V2 新增，取代原 projects.settlement_status】
        ├── project_id, user_id, settlement_status (unsettled/settling/settled)

space_types (空间类型字典表)
  └── id, name

construction_plans (施工方案字典表)
  └── id, name, unit(计量单位), price(单价)

advances (预支记录表)
  └── id, user_id, amount, date, remark, created_at

wage_settlements (工资结算单表)
  └── id, user_id, project_id, total_amount, advance_deducted, net_amount, settled_at

messages (站内消息表)
  └── id, user_id, title, content, type, read, created_at
```

## 🚫 已废弃/迁移的设计

| 旧设计 | 新设计 | 说明 |
|--------|--------|------|
| `projects.settlement_status` 字段 | `project_user_status.settlement_status` 表 | 改为按用户维度的独立结算状态，避免 A 用户结算后 B 用户看到错误状态 |
| `projects.settled_users` 字段 | 移除 | 数据冗余已消除 |
| `AuthInterceptor` 无自动刷新 | 401 时自动刷新 refresh_token | Android 端独立 OkHttpClient 调用 `/v1/auth/refresh` |
| 前端本地金额计算 | 统一调 `POST /v1/settlements/calculate` | 安全性与一致性 |
