3. 确保之前进行的"子项目内容缺失"、"工程历史不显示"等修复记录在案

**关键决策**：
- **AI 记忆方案**：建立 `.memory/` 目录，包含 5 个结构化文件
  - `记忆索引.md`：AI 对话入口（读取顺序 + 任务类型→文件映射）
  - `01-项目核心记忆.md`：项目定位、业务、角色权限、功能清单
  - `02-技术栈与架构.md`：三端技术栈、代码分层、数据库表结构
  - `03-开发规范与约束.md`：编码硬约束（金额计算、时区、安全、UI、构建验证）
  - `04-关键文件与快速定位.md`：所有核心文件路径索引 + 搜索关键词表
  - `05-历史会话摘要.md`：即本文件（对话成果追踪）
- **Token 刷新机制**：将 Android 端 `AuthInterceptor` 从"401 直接清除 token"改为"401 先自动调用 `/v1/auth/refresh` 刷新，失败再清 token"
- **后端 Token 有效期**：refresh_token 从 7d 延长到 30d
- **并发安全**：使用双重检查锁 + 同步块，防止多个 401 请求同时刷新 token
- **独立刷新客户端**：AuthInterceptor 使用独立 OkHttpClient，不依赖 Retrofit，避免循环依赖

**新增/修改的文件**：
| 文件 | 变更类型 | 说明 |

| `.memory/记忆索引.md` | 新建 | AI 对话入口（读取顺序 + 任务分类） |

| [salary-android/core/network/.../AuthInterceptor.kt](file:///f:/AIPoject/salary/salary-android/core/network/src/main/java/com/salary/core/network/interceptor/AuthInterceptor.kt) | 重写 | 从"401 直接清 token"改为"401 自动刷新 + 重试"，增加同步锁、独立 OkHttpClient、JSON 解析 |

**后端 auth.js 具体变更**：
- 第 11 行：`const REFRESH_TOKEN_EXPIRY = '7d'` → `'30d'`
- 登录路径（约 60 行处）：Redis TTL `7 * 24 * 60 * 60` → `30 * 24 * 60 * 60`
- 刷新路径（约 180 行处）：Redis TTL `7 * 24 * 60 * 60` → `30 * 24 * 60 * 60`
- 注册路径（约 255 行处）：Redis TTL `7 * 24 * 60 * 60` → `30 * 24 * 60 * 60`

**Android AuthInterceptor 重写核心逻辑**：
1. 从 `@Inject constructor(tokenStorage: TokenStorage)` 改为 `(tokenStorage: TokenStorage, serverConfig: ServerConfig)` —— 注入服务器配置
2. 请求流程不变：跳过 login/register/refresh 路径 → 读取 access_token → 添加 Authorization 头 → 发起请求
3. 收到 401 响应后的新流程：
   - 检查 token 是否已被其他请求刷新过（compare latest token with used token）
   - 进入同步块 `synchronized(refreshLock)`，使用双重检查锁
   - 调用 `performTokenRefresh()`：独立 OkHttpClient 直接请求 `POST {baseUrl}v1/auth/refresh`
   - 解析 `{ code, data: { accessToken, refreshToken } }` JSON 响应
   - 刷新成功：`tokenStorage.saveTokens(newAccess, newRefresh)` → 用新 token 重试原请求
   - 刷新失败（或正在刷新中）：等待最多 5 秒，其他请求刷新完成后用新 token 重试
   - 最终失败：`tokenStorage.clearTokens()`，AppNavHost 自动跳转登录页

### 2026-06-17 │ 工程历史时区修复 + 子项目金额事务修复 + Android 可空字段处理
**用户反馈的问题**：
1. "保存的工程没有子项目内容"
2. "主页工程历史里没有显示工程历史"

**根因分析**：
- **问题 1（子项目内容缺失）**：Android 端 `SubprojectDto` 中 `length`/`width`/`quantity`/`amount` 定义为非空 `Double`，但数据库允许 null，导致 kotlinx.serialization JSON 反序列化失败。同时后端在事务外计算金额后再 UPDATE，可能导致不一致。
- **问题 2（工程历史不显示）**：
  - 后端年月份筛选使用 UTC 时区，导致当天 00:00-08:00（CST = UTC+8）创建的工程被计入"上月"
  - 部分工程详情加载失败时，会导致所有工程从列表过滤掉，而非仅失败的工程被跳过

**修复内容**：

**后端**：
- `backend/repositories/projectRepo.js`：
  - `countQuery` 和 `listQuery` 中 `TO_CHAR(p.created_at, 'YYYY-MM')` → `TO_CHAR(p.created_at AT TIME ZONE 'Asia/Shanghai', 'YYYY-MM')`
  - `createProjectWithSubproject`：修改为直接使用传入的 `quantity` 和 `amount`，不再依赖事务外计算
- `backend/services/projectService.js`：
  - 金额计算移到事务内：先调用 `calculation.calculateSubprojectAmount(...)`，再将结果传入 `projectRepo.createProjectWithSubproject`
  - 删除单独的 `updateSubprojectAmount` 调用

**Android**：
- `core/network/src/main/java/com/salary/core/network/dto/ProjectDtos.kt`：`SubprojectDto` 中 `length`/`width`/`quantity`/`amount` 改为 `Double? = null`
- `feature/home/src/main/java/com/salary/manager/feature/home/dashboard/DashboardViewModel.kt`：
  - `mapToSubprojectModel`：可空字段用 `it.length ?: 0.0` 提供默认值，金额用 `AmountFormatter.format(it.amount)`
  - `loadProjects`：并行加载每个工程详情，失败时回退到"仅列表数据"的 `ProjectHistoryUiModel`（含子项目空列表），不再过滤掉失败的工程
- `feature/home/src/main/java/com/salary/manager/feature/home/detail/ProjectDetailViewModel.kt`：子项目映射同样处理可空字段
- `feature/home/src/main/java/com/salary/manager/feature/home/edit/ProjectEditViewModel.kt`：子项目编辑同样处理可空字段
